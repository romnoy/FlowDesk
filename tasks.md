# 📋 רשימת משימות לפיתוח - FlowDesk

רשימה זו מבוססת על מסמך ה-Spec וה-PRD של פרויקט FlowDesk. המשימות מסודרות בסדר לוגי כרונולוגי (מלמטה למעלה - מהבסיס לממשק המשתמש), כאשר כל משימה מוגדרת ליחידת עבודה של **20-30 דקות** ומכילה קריטריון הצלחה ברור ומדיד.

---

## 🏗️ אבן דרך 1: תשתית וסביבת עבודה

### [x] משימה 1.1: אתחול מבנה הפרויקט והתקנת תלויות (שרת ולקוח)
- **משך מוערך:** 25 דקות
- **תיאור:** יצירת תיקיות השרת והלקוח (`server` ו-`client`), אתחול קבצי `package.json` והתקנת תלויות נדרשות:
  - שרת: `express`, `cors`, `dotenv`, `node:sqlite` (מובנה), `multer`, `pdf-parse`, `mammoth`, `nodemon` (פיתוח)
  - לקוח: `react`, `react-router-dom`, `lucide-react`, `tailwindcss`, `@tailwindcss/postcss`, `postcss`, `autoprefixer`
- **קריטריון הצלחה:** הרצת `npm install` בשתי התיקיות ללא שגיאות, והגדרת סקריפטים להרצה.

### [x] משימה 1.2: הגדרת שלד שרת Express בסיסי ונקודת קצה לבדיקה
- **משך מוערך:** 20 דקות
- **תיאור:** יצירת קובץ השרת הראשי `server/index.js`, הגדרת CORS, JSON parsing, וטעינת משתני סביבה באמצעות `.env`. הגדרת נתיב בריאות בסיסי `GET /api/health`.
- **קריטריון הצלחה:** `verify:` שליחת בקשת `GET http://localhost:3001/api/health` מחזירה תשובת `{ "status": "ok" }` עם קוד 200.

### [x] משימה 1.3: הגדרת שלד לקוח React + Vite וחיבור ל-Tailwind CSS
- **משך מוערך:** 30 דקות
- **תיאור:** אתחול פרויקט React עם Vite בתיקיית `client`, התקנה והגדרה של Tailwind CSS ועיצוב בסיסי ב-`index.css`.
- **קריטריון הצלחה:** `verify:` הרצת `npm run dev` בלקוח מציגה דף בית ריק עם עיצוב כהה Sleek וגופן Outfit מודרני מ-Google Fonts.

## 🗄️ אבן דרך 2: בסיס נתונים (SQLite)

### [x] משימה 2.1: הגדרת סכמת הנתונים וכתיבת סקריפט אתחול (Setup DB)
- **משך מוערך:** 25 דקות
- **תיאור:** יצירת קובץ `server/db/schema.sql` המגדיר את הטבלאות `areas`, `projects`, `sub_projects`, `tasks`, ו-`transcripts` עם מפתחות זרים ותנאי CASCADE, וכתיבת קובץ `server/db/init.js` המאתחל את הקובץ `flowdesk.db` בעזרת `node:sqlite` המובנה.
- **קריטריון הצלחה:** `verify:` הרצת `node server/db/init.js` מייצרת קובץ `flowdesk.db` עם כל 5 הטבלאות תקינות ומוכנות לעבודה.

### [x] משימה 2.2: יצירת סקריפט Seed עם נתוני מוק (Mock Data) לבדיקה
- **משך מוערך:** 20 דקות
- **תיאור:** כתיבת סקריפט `server/db/seed.js` המכניס תחום ברירת מחדל `"כללי"`, מספר תחומי בדיקה (כגון "פיתוח", "שיווק"), פרויקטים, תתי-פרויקטים ומשימות דוגמה.
- **קריטריון הצלחה:** `verify:` הרצת הסקריפט מציגה הצלחה בקונסול, ושאילתה ידנית ל-DB מראה את כל הנתונים משויכים נכון היררכית.

## 🔗 אבן דרך 3: נתיבי API בשרת (Backend API Routes)

### [x] משימה 3.1: נתיבי CRUD עבור היררכיית פרויקטים (Areas, Projects, Sub-Projects)
- **משך מוערך:** 30 דקות
- **תיאור:** כתיבת נתיבי Express תחת `server/routes/hierarchy.js` לקריאה ויצירה של תחומים, פרויקטים ותתי-פרויקטים. הוספת וולידציה: מניעת שם כפול באותה רמת היררכיה עם השגיאה: `"שם זה כבר קיים ברמה זו. נא לבחור שם אחר."`.
- **קריטריון הצלחה:** `verify:` שליחת POST ליצירת פרויקט כפול תחת אותו תחום מחזירה סטטוס 400 עם הודעת השגיאה המדויקת בעברית.

### [x] משימה 3.2: מימוש מחיקה מדורגת (Cascade Delete) עם אימות
- **משך מוערך:** 25 דקות
- **תיאור:** מימוש נתיבי DELETE עבור תחומים ופרויקטים. הפעלת מנגנון `PRAGMA foreign_keys = ON;` בחיבור ה-SQLite כדי להבטיח שכל תתי-הפרויקטים והמשימות שתחתיהם יימחקו אוטומטית.
- **קריטריון הצלחה:** `verify:` מחיקת תחום עם משימות מוחקת את כל המשימות והפרויקטים המשויכים אליו בבסיס הנתונים (בדיקה על ידי שאילתה חוזרת).

### [x] משימה 3.3: נתיבי API לניהול משימות (Tasks CRUD)
- **משך מוערך:** 25 דקות
- **תיאור:** כתיבת נתיבי Express תחת `server/routes/tasks.js` ליצירה, קריאה, עדכון (PATCH) ומחיקה של משימות בבסיס הנתונים.
- **קריטריון הצלחה:** `verify:` שליחת PATCH לשינוי סטטוס משימה ל-"בביצוע" מעדכנת את הערך ב-DB ומחזירה קוד 200 עם המשימה המעודכנת.

## 📂 אבן דרך 4: העלאת קבצים ועיבוד מסמכים (Document Parsing)

### [x] משימה 4.1: מימוש נתיב העלאת קובץ (File Upload) וואלידציה
- **משך מוערך:** 25 דקות
- **תיאור:** הגדרת `multer` בשרת לקליטת קבצים בזיכרון (Memory Storage). מימוש וולידציה על סיומת הקובץ: מותר רק `.txt`, `.docx`, `.pdf`. אם לא תקין, החזרת שגיאה: `"פורמט הקובץ אינו נתמך, נא להעלות .txt, .docx או .pdf"`.
- **קריטריון הצלחה:** `verify:` העלאת קובץ `.png` מחזירה קוד 400 עם השגיאה המדויקת בעברית; העלאת קובץ `.txt` מחזירה קוד 200.

### [x] משימה 4.2: פיתוח מנגנון חילוץ טקסט בשרת (docx & pdf)
- **משך מוערך:** 30 דקות
- **תיאור:** כתיבת פונקציית עזר תחת `server/lib/parser.js` המשתמשת ב-`pdf-parse` עבור קבצי PDF וב-`mammoth` עבור קבצי Word לשאיבת טקסט נקי בזיכרון ללא שמירת הקובץ בדיסק.
- **קריטריון הצלחה:** `verify:` העברת Buffer של קובץ PDF או Word לפונקציה מחזירה מחרוזת טקסט מלאה המייצגת את תוכן המסמך.

### [x] משימה 4.3: יצירת רכיב Drop Zone להעלאת קבצים בלקוח
- **משך מוערך:** 30 דקות
- **תיאור:** בניית רכיב Upload Zone מעוצב ומודרני בלקוח התומך בגרירה והשלכה (Drag & Drop) של קבצים, כולל בדיקת סיומת בצד הלקוח והצגת הודעת שגיאה מתאימה.
- **קריטריון הצלחה:** `verify:` גרירת קובץ תקין מפעילה את פונקציית השילוח; גרירת קובץ לא תקין מציגה פופאפ אזהרה אדום ומונעת שליחה.

## 🤖 אבן דרך 5: אינטגרציה עם Claude API ועיבוד תמלולים

### [x] משימה 5.1: יצירת תמלול במצב 'processing' בבסיס הנתונים
- **משך מוערך:** 20 דקות
- **תיאור:** יצירת נתיב POST להעלאת תמלול אשר שומר רשומה ראשונית בטבלת `transcripts` עם סטטוס `'processing'` ומחזיר מיד `202 Accepted` עם ה-ID של התמלול תוך פחות מ-500ms.
- **קריטריון הצלחה:** `verify:` שליחת בקשת העלאה מחזירה מיד סטטוס 202 עם ה-ID של התמלול החדש, והרשומה ב-DB נמצאת במצב 'processing'.

### [x] משימה 5.2: הזרקת הקשר (Context Injection) ובניית הפרומפט ל-Claude
- **משך מוערך:** 30 דקות
- **תיאור:** כתיבת פונקציית יצירת פרומפט ב-`server/lib/ai.js`. הפונקציה שואבת את כל תחומי העניין, הפרויקטים ותתי-הפרויקטים הקיימים ומזריקה אותם להנחיות ה-AI כהקשר כדי שה-AI ישייך את המשימות שחולצו לפרויקטים הקיימים.
- **קריטריון הצלחה:** `verify:` הפרומפט המיוצר מכיל רשימה מדויקת של הפרויקטים הקיימים במערכת ומנחה את ה-AI להחזיר JSON בלבד בהתאם לחוקי ה-Skill.

### [x] משימה 5.3: מימוש קריאה ישירה ל-Claude API ועדכון הסטטוס ב-DB
- **משך מוערך:** 30 דקות
- **תיאור:** מימוש הקריאה ל-API של Claude באמצעות `fetch` ישיר (ללא SDK). הטיפול בתוצאה כולל ניקוי של תגיות Markdown JSON, חסכון בקריסות (try/catch), ושמירת התוצאה המובנית ב-DB תוך שינוי סטטוס ל-`completed` או `failed` בהתאמה.
- **קריטריון הצלחה:** `verify:` הרצת התהליך בהצלחה משנה את הסטטוס ב-DB ל-`completed` ושומרת JSON מובנה בשדה `structured_analysis`.

- **קריטריון הצלחה:** `verify:` סרגל הצד נטען ומציג את ההיררכיה המדויקת מה-DB עם אפשרות לפתיחה/סגירה של פרויקטים.

### [x] משימה 6.2: בניית מסך הדשבורד (Dashboard) ומצבי אפס (Zero State)
- **משך מוערך:** 25 דקות
- **תיאור:** עיצוב מסך הבית הראשי של המערכת המציג כרטיסי סיכום: סה"כ משימות פתוחות, משימות בעדיפות גבוהה, ומשימות לשבוע הקרוב. הוספת מצב אפס: `"אין משימות כרגע"`.
- **קריטריון הצלחה:** `verify:` כאשר אין משימות ב-DB, מוצגת הודעת מצב האפס המעוצבת יחד עם קישור להעלאת תמלול; כשיש משימות, הנתונים הסטטיסטיים נכונים ומדויקים.

### [x] משימה 6.3: בניית דיאלוגים (Modals) ליצירה ומחיקה של פרויקטים/תחומים
- **משך מוערך:** 25 דקות
- **תיאור:** מימוש חלוניות קופצות (Modals) מעוצבות ליצירת תחום/פרויקט חדש כולל הצגת שגיאות וולידציה כפולות, ודיאלוג אישור מחיקה מבוסס CASCADE.
- **קריטריון הצלחה:** `verify:` ניסיון ליצור פרויקט כפול מציג את הודעת השגיאה הנדרשת בתוך המודאל באופן אדום ובולט; מחיקה מבקשת אישור.

## 📋 אבן דרך 7: לוח קנבן (Kanban Board) וניהול משימות

- **משך מוערך:** 30 דקות
- **תיאור:** יצירת עמוד פרויקט המכיל לוח קנבן עם שלוש עמודות קבועות בדיוק: `"טרם התחיל"`, `"בביצוע"`, `"הושלם"`. בניית כרטיס משימה הכולל כותרת, עדיפות, תאריך יעד, וסימון אדום בולט עם אייקון התראה למשימה שפג תוקפה ואינה בסטטוס "הושלם".
- **קריטריון הצלחה:** `verify:` משימות מוצגות בעמודה המתאימה; משימה שתאריך היעד שלה עבר ואינה מושלמת מקבלת גבול אדום ואייקון אזהרה בולט.

### [x] משימה 7.2: מימוש גרירה ושחרור (Drag and Drop) עם עדכון אופטימי (Optimistic UI)

- **קריטריון הצלחה:** `verify:` הדמיית כשל בשרת מחזירה את המשימה לעמודה המקורית במהירות הבזק ומציגה הודעת שגיאה ברורה למשתמש.

### [x] משימה 7.3: דיאלוג פרטי משימה ויצירה/עריכה ידנית
- **משך מוערך:** 25 דקות

- **קריטריון הצלחה:** `verify:` לחיצה על משימה פותחת את המודאל, שינוי עדיפות ושמירה משתקפים מיד בלוח ובבסיס הנתונים.

## 🔍 אבן דרך 9: ארכיון פגישות וחיפוש

### [x] משימה 9.1: דף ארכיון וחיפוש טקסט מלא (SQLite Full-Text Search)
- **משך מוערך:** 30 דקות
- **תיאור:** יצירת דף ארכיון המרכז את כל התמלולים ההיסטוריים. מימוש שורת חיפוש המבצעת שאילתת SQL הכוללת חיפוש טקסט חופשי (LIKE או FTS) בשם התמלול, בתוכן הגולמי ובתיאור המשימות המשויכות אליו.
- **קריטריון הצלחה:** `verify:` הזנת מילת מפתח מסננת את רשימת התמלולים במהירות ומציגה רק את אלו המכילים את המילים שהוזנו. בהיעדר תוצאות מוצג: `"לא נמצאו תמלולים התואמים לחיפוש"`.
